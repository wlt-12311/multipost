# 宇之然「云帆」AI 营销平台 — 产品原型 & 技术实施方案

> 产品代号：**云帆**（云帆济沧海，帮产品出海）
> 域名：yuzhiran.com
> 定位：给独立开发者和一人公司的 AI 营销工作台

---

## 一、产品总览

### 一句话
输入你的产品/项目 → AI 生成营销策略+内容+页面 → 展示在营销广场获得曝光

### 三层架构回顾
```
┌──────────────────────────────────────────┐
│  第三层：营销广场                          │
│  所有用户页面汇聚 + 分类浏览 + 互推联盟    │ ← 网络效应
├──────────────────────────────────────────┤

│  个人页/公司页/产品页 + 托管/导出          │ ← 收费层

└──────────────────────────────────────────┘
```

## 二、核心功能清单（逐项展开）

### 2.1 第一层：AI 引擎

#### F1-策略分析
用户输入：
- 产品/项目名称（必填）
- 一句话描述（必填）
- 所属行业（选填，自动识别）
- 目标用户（选填）
- 竞争对手（选填）
- 当前阶段（想法/开发中/已上线/已有用户）

AI 输出（三级深度）：

| 级别 | 耗时 | 输出内容 |
|:---:|:----:|:--------|
| ⚡ 速览 | ~10s | ①定位语 ②3个核心卖点 ③建议首发渠道 ④一句话受众描述 |
| 📋 标准 | ~30s | ①定位语+品牌调性建议 ②完整受众画像 ③3-5个竞品概览 ④渠道建议+优先级排序 ⑤内容日历大纲 |
| 🎯 深度 | ~60s | ①完整定位方案 ②受众调研报告 ③SWOT分析 ④定价策略建议 ⑤冷启动计划 ⑥前30天内容日历 ⑦SEO关键词规划 |

技术实现：通过 AI 网关路由到 Claude（深度推理），使用链式 Prompt（先分析→再拆解→再输出结构化 JSON）。

#### F2-内容工厂
用户选择：
- 目标渠道（多选：Twitter、LinkedIn、知乎、V2EX、ProductHunt、Email、README）
- 语气风格（专业/朋友/故事/教育/促销）
- 语言（中文/英文）

输出：
| 渠道 | 内容格式 | 长度 |
|:-----|:--------|:----|
| Twitter/X | Thread（编号1/n） | 每条≤280字符，8-12条 |
| LinkedIn | 叙事长文 | 800-2000字 |
| 知乎 | 问答式/文章 | 1000-3000字 |
| V2EX | 简洁介绍帖 | 200-500字 |
| ProductHunt | 标题+标语+描述+评论模板 | 标题≤60字符 |
| Email | Subject + Body | 300-800字 |
| GitHub README | 完整README markdown | 标准长度 |

每个渠道输出可：
- 一键复制
- 重新生成（换语气/换角度）
- 手动编辑后保存

#### F3-广场卡片
内容工厂自动生成当前产品的「广场展示卡片」内容：
- 产品名 + Logo占位
- 一句话标语
- 标签分类
- 价格（如有）
- 创作者信息

## 三、用户角色与流程

### 3.1 用户角色

| 角色 | 权限 |
|:----|:----|
| 访客 | 浏览广场、浏览用户页面、注册 |

| Pro用户 | 页面托管、多语言、广场展示、自定义域名×1 |

### 3.2 核心流程

#### 新用户完整路径
```
注册/登录
  → 进入工作台
  → "创建一个新页面"
  → 填写产品信息
  → AI 策略分析（可选深度）
  → AI 内容生成（选渠道）
  → 选择页面模板
  → AI 生成页面
  → 手动编辑调整
  → 发布（托管或导出）
  → 页面自动进广场
  → 在后台开启互推
```

#### 访客路径
```
访问 yuzhiran.com/plaza
  → 浏览卡片
  → 点击 → 进入用户产品页
  → 底部看到互推推荐
  → 点击互推 → 另一个产品页
  → "我也想要一个？" → 注册
```

### 5.3 AI 生成流程

```
用户请求生成页面
  ↓
策略引擎：收集信息 → 调用 AI（Claude/GPT-4）→ 结构化策略输出
  ↓
内容工厂：基于策略 → 并行调用 AI（DeepSeek）→ 各渠道内容
  ↓
页面生成器：基于模板 → 填充内容 → 调用 AI（DeepSeek）→ 微调文本 → 输出页面 HTML
  ↓
多语言翻译：调用 AI（DeepSeek）→ 翻译各语言版本
  ↓
保存到数据库 / 生成静态文件
```

**每次 AI 调用都通过 AI 网关进行，自动路由到最优模型。**

### 5.4 数据库设计（核心表）

            output(json), created_at

          content(text), tone, lang, version, created_at

       slug(unique), config(json), published(bool), 
       custom_domain, created_at, updated_at

-- 页面访问统计

-- 互推关系
referrals: id, from_page_id, to_page_id, active(bool), created_at

-- 广场卡片

             sort_order, created_at

-- 订阅
subscriptions: id, user_id, plan, status, 
               start_date, end_date, payment_provider

## 八、关键决策记录

| 决策 | 选择 | 理由 |
|:----|:----|:-----|
| 页面渲染方式 | API驱动（React渲染） | MVP 阶段简化，无需 SSG/SSR |
| 多语言实现 | react-i18next | 成熟方案，支持 lazy loading |
| 页面托管 | 后端动态渲染 | 避免静态文件管理复杂度 |
| 广场搜索 | 数据库 LIKE/全文检索 | MVP 阶段简单实现，后期可上 Elasticsearch |
| 互推匹配 | 手动选择 + 系统推荐 | 先让用户自己选，AI推荐后续迭代 |
| 容器化 | Docker Compose | 统一开发/生产环境 |

## 九、风险备忘录

| 风险 | 程度 | 应对 |
|:----|:----|:-----|
| AI 输出质量不稳定 | 高 | 持续迭代 prompt；给用户"重新生成"选项 |
| 广场初期没内容（冷启动） | 高 | 我们自己先用产品生成5-10个示范页面；邀请朋友填充 |
| 用户感觉不如免费ChatGPT | 中 | 强调"结果导向"而非"对话体验"；展示广场曝光价值 |
| PostgreSQL 运维成本 | 低 | 初期用 SQLite 过渡，用户数<1000 不用切 |
| 海外 Stripe 收款问题 | 中 | 初期只做国内支付（支付宝），国际支付走网关额度模式 |
